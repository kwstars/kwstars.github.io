Garbage Collection Semantics

2021年6月7日
| Go
| 阅读

从1.12版本开始，go使用了一个非生成的、并发的、三色的、标记和清扫的收集器

Mark Setup

当一个垃圾收集开始时，必须执行的第一个活动是打开写屏障。

为了打开写屏障，每一个正在运行的应用程序的程序都必须停止。

在垃圾收集开始之前，有四个应用程序的程序在运行。

在1.14中修复，但现在哦我们正在等待

Marking

检查堆栈以找到指向堆的根指针。 从这些根指针遍历堆图。 标记堆上仍在使用中的值。

采集器将25%的可用CPU容量占为己有。

放慢分配速度，反过来加快收集速度。 采集器的目标是消除对标记辅助的需要。 如果一个收集需要大量的标记辅助，收集器可以提前开始下一个收集。

Mark Termination

关掉写入障碍。 执行各种清理任务。 计算下一个收集目标。

Collector Behavior

Sweeping

扫荡

释放堆内存

当goroutines试图分配新的堆内存时发生。 扫频的延迟被添加到执行分配的成本中，而不是GC。

当新的分配出现时，扫除内存。

GC Percentage

开始收集

默认情况下，GC百分比被设置为100。 代表在下一个集合开始之前可以分配多少新的堆内存的比例。 用GOGC环境变量设置。

免责声明

这张牌中的堆内存图并不代表使用围棋时的真实情况。 围棋中的堆内存往往是零散的、混乱的，你不可能像图片中表现的那样，有一个干净的分离。 这些图表提供了一种可视化堆内存的方法，准确地反映了这种行为。

算法 有一个节奏算法，用于确定何时开始收集。 反馈环路，收集有关运行中的应用程序和压力的信息。 压力，决定采集器需要运行的速度。
